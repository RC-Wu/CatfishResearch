# Catfish Architecture Roadmap

## 1. Objective

This roadmap turns the Catfish architecture into an implementation sequence for this repository's current style: docs first, portable tools second, larger services only when the contracts are stable.

## 2. Phase 0: Lock The Contracts

Deliverables:

- architecture doc
- scoring and competition doc
- multi-project runtime doc
- roadmap
- schemas for capability metadata, scorecards, and scheduling graph
- repo-local `agent.md`

Exit criteria:

- terminology is stable
- parent-only scoring is specified clearly enough to code against
- capability metadata fields are machine-readable
- scheduling graph fields are concrete enough for a first state object

## 3. Phase 1: Minimal Runtime Substrate

Implementation targets:

- `tools/catfish/manifests.py`
- `tools/catfish/capability_registry.py`
- `tools/catfish/scorecards.py`
- `tools/catfish/state.py`

Scope:

- load project manifests
- parse `Capability.md` front matter
- validate and write scorecards
- persist scheduling-graph snapshots

Exit criteria:

- one project can be represented fully on disk
- one parent can write scorecards for several child candidates
- scorecards fail validation if the writer is not the parent

## 4. Phase 2: Competition Runtime

Implementation targets:

- `tools/catfish/competition.py`
- `tools/catfish/policies.py`

Scope:

- generic CompetitionCell execution
- stage-specific rubric loading
- winner-take-all, top-k, merge-then-re-score, and replay-under-constraint-shift modes

Exit criteria:

- ideation and figure-generation competitions run through the same runtime contract
- advancement decisions are replayable from stored artifacts and scorecards

## 5. Phase 3: Scheduling Graph And Negotiation

Implementation targets:

- `tools/catfish/scheduler.py`
- `tools/catfish/resource_market.py`

Scope:

- cross-project frontier management
- per-project and per-team slot budgeting
- MCTS-like selection heuristic with priors and realized rewards

Exit criteria:

- scheduler can rank several project expansion requests
- graph state can explain why a branch was expanded or pruned
- budget and provider constraints are first-class inputs

## 6. Phase 4: Self-Update Loops

Implementation targets:

- `tools/catfish/update_loops.py`
- `tools/catfish/postmortems.py`

Scope:

- project-level prior updates
- team-level role-mix updates
- system-level policy proposals

Exit criteria:

- updates are versioned
- updates are evidence-backed
- system-level changes do not silently rewrite project history

## 7. Phase 5: Visualization And Reporting

Implementation targets:

- `tools/catfish/visualization.py`
- `tools/catfish/exporters.py`

Scope:

- system hierarchy figure export
- score breakdown charts
- scheduling frontier snapshots
- grayscale-safe, paper-friendly rendering defaults

Exit criteria:

- one project graph and one system graph can be exported from stored state
- figures are clean enough for design review and eventual paper inclusion

## 8. Phase 6: Runtime Integration

Implementation targets:

- optional CLI entrypoints under `tools/`
- example manifests under `tools/examples/catfish/`
- tests under `tools/tests/`

Scope:

- lightweight end-to-end flow from manifest to competition to scorecard to visualization
- no heavy web control plane required yet

Exit criteria:

- local replay of a small multi-stage project works from disk artifacts alone
- the repo remains portable and auditable

## 9. Validation Strategy

Cheap validation for the current phase:

- JSON schema files parse cleanly
- markdown docs cross-reference the same core entities
- git diff is limited to owned files

Stronger validation for later phases:

- schema-backed manifest tests
- scorecard parent-authorship tests
- scheduling selection tests
- visualization snapshot tests

## 10. Risks And Countermeasures

Risk:

- overbuilding a control plane before the contracts are stable

Countermeasure:

- keep the first implementation file-backed and replayable

Risk:

- allowing one score dimension to dominate all others

Countermeasure:

- persist full raw dimensions and stage-specific weights

Risk:

- collapsing onto one provider-model stack too early

Countermeasure:

- preserve diversity bonus and replay-under-constraint-shift mode

Risk:

- letting self-update become silent drift

Countermeasure:

- version all policy updates and tie them to evidence references

## 11. Immediate Next Step

The next concrete coding task after this design pass should be Phase 1: implement the file-backed contract layer and add unit tests for capability parsing, scorecard validation, and scheduling-graph serialization.
