# Catfish Control Center Skeleton

This page describes the original backend-first skeleton. The current live-state baseline is documented in [`CONTROL_CENTER_LIVE_20260325.md`](CONTROL_CENTER_LIVE_20260325.md), and the code now supports both `--snapshot` and `--state-root` inputs.

## Purpose

This repository already had a lightweight routing core in `tools/codex_route_preview.py`, but it explicitly stopped short of a persistent control-plane surface. `apps/catfish-control-center` is the missing next layer for CatfishResearch: a backend-first control-center skeleton that can materialize multi-project state, display agent hierarchy, track provider health and quota pressure, compare competing branches, and preserve both snapshot and append-only event views.

The goal is practical scaffolding, not a premature product UI. The current shape is intentionally CLI-first so Catfish can start emitting and inspecting control-plane state before choosing a web or TUI frontend.

## What Was Added

- `apps/catfish-control-center/main.py`
  - thin entrypoint that runs the control-center CLI
- `apps/catfish-control-center/catfish_control_center/models.py`
  - typed domain objects for projects, agents, providers, branch scoreboards, events, and whole control snapshots
- `apps/catfish-control-center/catfish_control_center/storage.py`
  - storage interfaces plus JSON snapshot and JSONL event adapters
- `apps/catfish-control-center/catfish_control_center/runtime.py`
  - runtime helpers that load snapshots, merge events, and enrich provider selection via the existing route-preview tool
- `apps/catfish-control-center/catfish_control_center/dashboard.py`
  - text dashboard renderer for overview, hierarchy, provider state, scoreboards, and recent events
- `apps/catfish-control-center/examples/sample_snapshot.json`
  - concrete Catfish-shaped sample state
- `apps/catfish-control-center/tests/test_control_center.py`
  - cheap validation for rendering, route-preview integration, and storage adapters

## Runtime Fit With Existing Architecture

The control center is designed to sit next to the current Codex routing core, not replace it.

1. `tools/codex_route_preview.py` remains the routing authority for profile and tier selection.
2. Catfish runtime code can materialize a `ControlSnapshot` from live execution state.
3. The control center can optionally apply route preview against `tools/examples/control_plane.example.json` or a future Catfish-specific config.
4. Selected provider state is reflected in the dashboard without changing routing behavior.
5. Event feeds can be appended as JSONL, giving Catfish an append-only log even before it adopts a database or message bus.

That means the current skeleton is compatible with:

- one-shot state dumps generated by orchestration scripts
- heartbeat and monitor agents that append events
- future promotion to SQLite, Postgres, Redis streams, or an event bus by swapping storage adapters behind the same interfaces

## Data Model

The control center keeps the model deliberately small:

- `ProjectState`
  - status of each Catfish project or wave
- `AgentNode`
  - hierarchy-ready representation of orchestrators, workers, reviewers, and monitors
- `ProviderState`
  - machine reachability, identity health, quota headroom, routing weight, and supported tiers
- `BranchScore`
  - branch-vs-branch competition state for scoreboard views
- `ControlEvent`
  - append-only event records for heartbeat, quota, branch-score, review, and runtime transitions
- `ControlSnapshot`
  - materialized view consumed by the CLI or any future UI

## Storage Contract

Two storage interfaces are enough for the first usable control-center:

- `SnapshotStore`
  - load and save the materialized state view
- `EventStore`
  - append and query recent runtime events

Current adapters:

- `JsonSnapshotStore`
  - simplest materialized-state backend for generated snapshots
- `JsonLinesEventStore`
  - append-only event backend for cheap local persistence
- `InMemoryEventStore`
  - testing and integration glue

This is intentionally the smallest shape that still gives Catfish a clean path from file-backed state to a real service backend.

## Practical Usage

Render the sample dashboard:

```bash
python apps/catfish-control-center/main.py \
  --snapshot apps/catfish-control-center/examples/sample_snapshot.json \
  --control-plane-config tools/examples/control_plane.example.json \
  --machine dev-intern-02 \
  --task-kind builder \
  --difficulty high
```

Dump the merged state as JSON instead:

```bash
python apps/catfish-control-center/main.py \
  --snapshot apps/catfish-control-center/examples/sample_snapshot.json \
  --format json
```

## How Catfish Should Connect Next

The intended runtime wiring is:

1. Catfish orchestrators emit heartbeat, provider, review, and branch-score events into an `EventStore`.
2. A lightweight aggregator periodically materializes those events plus live project metadata into a `ControlSnapshot`.
3. The CLI consumes the snapshot directly for local inspection.
4. If Catfish later needs a browser UI, that UI should read the same snapshot and event abstractions instead of inventing a parallel model.

This keeps the routing core stable, avoids touching unrelated routing-worker files, and gives CatfishResearch a concrete place to grow a real control center from.
